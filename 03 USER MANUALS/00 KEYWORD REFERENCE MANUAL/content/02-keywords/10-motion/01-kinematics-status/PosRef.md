# PosRef

Position reference into the position loop, after profiler post-processing.

## Overview

`PosRef` is the position reference, in main user units, after the optional post-processing steps (offset, moving average, input shaping, injection and filter) applied to the value generated by the motion profiler. It is the input to the position loop: the position error [PosErr](PosErr.md) is `PosRef − Pos`, and the velocity reference [dPosRef](dPosRef.md) is its filtered derivative.

`PosRef` is read-only. It is not a single number but the *reported* end of a multi-stage reference pipeline that the firmware maintains internally at higher precision (a 50.14 fixed-point accumulator) so that fractional motion accumulates without drift.

## How it works

### Reference pipeline

The profiler produces a raw target each control cycle; that value is then post-processed through several internal stages before it reaches the loop. The firmware carries each stage as a 64-bit fixed-point variable scaled by `2^SAMPLE_FREQUENCY_TWO_POWER` (`= 2^14`), and `PosRef` is reported by shifting the **shaped** reference back down to user units:

```text
profiler ─► gllPosRef ─► (offset / moving-average) ─► gllPosRefSmooth
         ─► [input shaping] ─► gllPosRefShaped ─► [injection] ─► [filter] ─► gllPosRefShapedFilt ─► position loop
```

`PosRef` is reported from the shaped stage as `glPosRef = gllPosRef >> 14` (e.g. `AG300_CTL01Profiler.c:877`, `1528`), with rounding (`+ GLLPOSREF_HALF`) at some sites. The fully post-processed value that the position loop actually subtracts the feedback from is `gllPosRefShapedFilt` (used as `glPosRefShapedFilt` in `AG300_CTL01ControlLoops.c:424`), so the reported `PosRef` is one snapshot of this chain rather than the literal loop input after shaping/filtering.

### Software position-limit clamping

When the reference exceeds a software travel limit it is clamped to the limit rather than reported beyond it: `glPosRef = glFwdPLim` / `glPosRef = glRevPLim` (e.g. `AG300_CTL01Profiler.c:1198`, `1203`, `2949`). See [FwdPLim](../../06-protections/03-motion/position-limit-protection/FwdPLim.md) and [RevPLim](../../06-protections/03-motion/position-limit-protection/RevPLim.md).

### Motor-off and simulation

While the motor is off the controller forces the reference to track the live feedback — `glPosRef = glPos` (`AG300_CTL01Profiler.c:10337`, `10502`) — so position error is zero at the instant of enable and there is no jump. In **simulation** (`MotorType` = simulation, value 5) the firmware feeds the shaped reference back in as the encoder reading (`*lpMainEncPos = glPosRefShapedFilt`, `AG300_CTL01ControlInterrupt.c:2237`), so the feedback [Pos](Pos.md) follows `PosRef` exactly.

### Modulo (ModRev)

If [ModRev](../../03-encoder/04-modulo-mode/ModRev.md) ≠ 0, when the feedback wraps the firmware shifts the **whole reference frame** by `ModRev` in the same control cycle — `gllPosRef`, `gllPosRefSmooth`, `gllPosRefShaped` and all of the shaped/filtered history variables are offset together (`AG300_CTL01ControlInterrupt.c:3172`–`3177`) — so the following error is preserved across the wrap and `PosRef` stays inside the modulo frame.

## Examples

```text
APosRef             ; read the current position reference
```

## Changes between versions

In **v5 (central-i)** the pipeline is 64-bit (`PosRef` is reported as a 64-bit value with the larger range shown in the frontmatter); the reference stages and clamping behaviour are the same. **v5 is central-i only**, so on standalone `PosRef` remains the v4 32-bit value.
